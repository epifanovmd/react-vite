# Архитектура и конвенции

Документ описывает структуру проекта, правила размещения кода и именования
папок/файлов. Цель — единый предсказуемый стиль: новый человек видит один
паттерн во всём репозитории.

---

## 1. Слои проекта

```
src/
├── api/            # сгенерированный API-клиент (swagger) + ApiService (axios, интерсепторы)
├── components/     # UI: дизайн-система, каркас, переиспользуемые компоненты
│   ├── ui/         #   примитивы БЕЗ домена (Button, Card, Modal, InfoField…)
│   ├── layouts/    #   структурный каркас (PageLayout, Header, AuthLayout, AuthFormCard)
│   ├── shared/     #   доменные переиспользуемые (UserAvatar, PageLoader)
│   └── slots/      #   утилиты композиции
├── core/           # инфраструктура приложения (auth, env, notifications, socket)
├── di/             # IoC-контейнер (Inversify) + декораторы сервисов
├── hooks/          # глобальные переиспользуемые хуки
├── pages/          # экраны приложения (см. раздел 3)
├── routes/         # файловый роутинг TanStack Router (тонкие обёртки над pages)
├── store/          # состояние (MobX): сторы, холдеры, модели
│   ├── holders/    #   обёртки async-состояния (Entity/Paged/Collection/…)
│   ├── models/     #   модели-обёртки над DTO (computed-геттеры)
│   ├── <domain>/   #   сторы по доменам (auth, app)
├── utils/          # чистые утилиты
├── App.tsx         # корень приложения
└── router.tsx      # конфигурация роутера
```

### Псевдонимы путей (alias)

Импорт между слоями — **только через алиасы**, не через относительные `../../..`:

| Alias          | Путь                |
| -------------- | ------------------- |
| `@components`  | `src/components`     |
| `@store`       | `src/store`         |
| `@models`      | `src/store/models`  |
| `@api`         | `src/api`           |
| `@core`        | `src/core`          |
| `@di`          | `src/di`            |
| `@hooks`       | `src/hooks`         |
| `@utils`       | `src/utils`         |

Относительные импорты допустимы **только внутри одной фичи/страницы** (между
её под-папками: `../profile-details`, `./hooks`).

## 2. Лестница продвижения компонентов

Компонент живёт на **самом низком уровне, который покрывает его фактическое
использование**. Продвигаем вверх только при **реальном** переиспользовании
(правило «трёх»), не заранее.

| Уровень                      | Что туда кладём                              | Критерий                       |
| ---------------------------- | -------------------------------------------- | ------------------------------ |
| `pages/<page>/components/`   | композиты конкретной страницы                | используется **одной** страницей |
| `components/shared/`         | доменные переиспользуемые (знают про «юзера») | используется **≥2** страницами  |
| `components/ui/`             | примитивы дизайн-системы, **без домена**      | generic, переиспользуется везде |
| `components/layouts/`        | структурный каркас                            | обрамление страниц             |

Правила:

- **`ui/` не знает о домене** — там нет слов «профиль», «пользователь».
  Если компонент знает доменное понятие, но переиспользуется — это `shared/`.
- **Одноразовое не поднимаем** в `shared`/`ui` — оно остаётся page-local.
- **Нет мёртвого кода** и осиротевших барелей: удалил фичу — удали её
  компоненты/сторы/модели.

## 3. Структура страницы (`pages/<page>`)

Структура страницы растёт вместе с её размером. Не задавай вложенность авансом.

### Архетипы

| Размер страницы                    | Архетип                                                    |
| ---------------------------------- | --------------------------------------------------------- |
| 1 экран, ~нет логики               | **1 файл** (`ErrorPage.tsx` + `index.ts`)                 |
| 1 форма/логика, без под-блоков     | **плоский** + `validations.ts` + `hooks/useXxxVM.ts`      |
| логика + 2–3 блока                 | **группировка по типу** (`hooks/`, `components/`, `modals/`) |
| крупные самостоятельные блоки      | **папка-на-компонент** внутри `components/`                |
| автономная фича со своими данными  | **вертикальный срез** (`api/ model/ ui/ lib/`)            |

### Эталон (папка-на-компонент)

```
pages/profile/
├── Profile.tsx                 # View: тонкий контейнер (observer), VM → пропсы
├── index.ts                    # barrel: export { Profile }
├── hooks/                      # ViewModel страницы
│   ├── index.ts
│   └── useProfileVM.ts
└── components/                 # презентационные блоки страницы
    ├── index.ts
    ├── profile-card/
    │   ├── ProfileCard.tsx
    │   ├── ProfileCard.types.ts
    │   └── index.ts
    └── edit-profile-modal/     # блок со своей логикой:
        ├── EditProfileModal.tsx
        ├── EditProfileModal.types.ts
        ├── useEditProfileVM.ts #   VM колокирован с компонентом
        ├── validations.ts      #   схема колокирована с компонентом
        └── index.ts
```

Правила:

- **`components/` заводим по факту** появления реальных композитов, а не заранее.
  Один маленький блок — оставляем в файле страницы или внутри родителя.
- **Каждый блок — своя папка** с `Component.tsx` + `index.ts`; `Component.types.ts`
  при нетривиальных пропсах; `useXxx.ts`/`validations.ts` — рядом, если у блока
  своя логика/валидация.
- Несколько экранов в одной фиче (`pages/auth/sign-in`, `pages/auth/sign-up`) —
  каждый экран это своя папка-«страница» со своими `hooks/`, `components/`.

## 4. MVVM и Container / Presentational

Слои разделяем по ответственности:

- **View (контейнер)** — `*.tsx` страницы/экрана. Тонкий: читает VM, раздаёт
  данные вниз. Оборачивается в `observer`, если читает MobX-обсервабли.
- **ViewModel** — хук `useXxxVM` в `hooks/`. Вся логика: формы, загрузка,
  сайд-эффекты, обработчики. Возвращает `form` + `handleXxx`/`submit` + данные.
- **Model** — стор/модель в `src/store`. Состояние и доменная логика.

Презентационные компоненты:

- **не знают о сторе** — получают данные и колбэки **через пропсы**
  (легко тестировать и переиспользовать);
- маппинг `модель → пропсы` делает контейнер (читает обсервабли, передаёт примитивы);
- `observer` вешаем только там, где реально читаются обсервабли.

## 5. Именование

### Папки — `kebab-case`

```
components/ui/info-field/
components/layouts/auth-layout/
pages/reset-password/
pages/auth/sign-in/
pages/profile/components/profile-card/
```

### Файлы

| Тип файла                | Стиль          | Пример                       |
| ------------------------ | -------------- | ---------------------------- |
| Компонент React          | `PascalCase`   | `ProfileCard.tsx`            |
| Типы компонента          | `PascalCase.types.ts` | `ProfileCard.types.ts` |
| Хук / VM                 | `camelCase`    | `useProfileVM.ts`            |
| Стор / сервис            | `PascalCase.<role>.ts` | `Auth.store.ts`, `Api.service.ts` |
| Типы стора               | `PascalCase.types.ts` | `Auth.types.ts`        |
| Модель                   | `PascalCase.model.ts` | `Profile.model.ts`     |
| Схемы валидации          | `validations.ts`      |                        |
| Барель                   | `index.ts`            |                        |
| Утилиты                  | `camelCase.ts`        | `lambdaValue.ts`       |

**Папка ≠ файл по casing:** папка `profile-card/`, файл внутри `ProfileCard.tsx`.
Имя файла компонента = имени экспортируемого компонента = имени папки (в kebab).

### Сущности в коде

- Компоненты: `PascalCase` (`ProfileCard`).
- Хуки: `useCamelCase`, VM — `useXxxVM`.
- Типы/интерфейсы: `PascalCase`; пропсы — `XxxProps`.
- Zod-схемы: `xxxFormValidationSchema`; выведенные типы — `TXxxForm`.
- Сторы (DI): класс + интерфейс `IXxxStore`, декоратор `IXxxStore`.

## 6. Барели (`index.ts`)

- Каждая папка-компонент, `hooks/`, `components/`, слой — экспортируется через
  `index.ts`. Снаружи импортируем из барреля, не из конкретного файла:
  `import { ProfileCard } from "./components"`, `import { Button } from "@components/ui"`.
- Барель компонента реэкспортит сам компонент и его публичные типы:
  ```ts
  export { ProfileCard } from "./ProfileCard";
  export type { ProfileCardProps } from "./ProfileCard.types";
  ```
- Барель скрывает внутреннюю структуру папки — переименование/перестановка
  файлов внутри не задевает потребителей.

---

## 7. Состояние (MobX + DI)

- **Сторы** — классы-синглтоны, регистрируются в IoC-контейнере (`@di`),
  инжектятся через декораторы. В компонент попадают через хук `useXxxStore`
  (обёртка `iocHook`).
- **Холдеры** (`store/holders`) — переиспользуемые обёртки async-состояния
  (статусы загрузки, ошибки, отмена): `EntityHolder`, `PagedHolder`,
  `CollectionHolder`, `MutationHolder`, `InfiniteHolder`, `PollingHolder`.
- **Модели** (`store/models`) — обёртки над DTO с `computed`-геттерами для UI.
- **Серверное и клиентское состояние не смешиваем** в одном месте без причины:
  серверные данные тянем через `ApiService` в холдеры; UI-флаги — локально/в сторе.
- Компонент, читающий обсервабли, обязан быть `observer` (`mobx-react-lite`).

## 8. Чеклист для новой страницы

1. Завести `pages/<page>/` (kebab). Маршрут — тонкая обёртка в `routes/`.
2. View-контейнер `Page.tsx` + `index.ts` (barrel).
3. Логику — в `hooks/usePageVM.ts`. Контейнер только связывает VM и UI.
4. Схемы — в `validations.ts` (`xxxFormValidationSchema` / `TXxxForm`).
5. Сложные блоки — в `components/<block>/` (папка-компонент + `.types.ts`).
6. Использовать готовые `@components/ui`; повторяющийся паттерн → поднять в `ui`/`shared`
   или создать новый примитив.
7. Проверить: `tsc --noEmit`, `eslint`, `vite build` — зелёные; мёртвого кода нет.

---

## 9. Чеклист для нового UI-компонента

1. Доменно-нейтральный примитив → `components/ui/<kebab>/`;
   доменный переиспользуемый → `components/shared/<kebab>/`;
   одноразовый для страницы → `pages/<page>/components/<kebab>/`.
2. Папка: `Component.tsx` + `index.ts` (+ `Component.types.ts`, `componentVariants.ts` при необходимости).
3. Стили — Tailwind + токены темы (`text-foreground`, `bg-card`, `border-border`),
   **без хардкода цветов** (`bg-red-500` запрещён, используем `destructive`/`success`/…).
4. Вариативность — через `cva` (`class-variance-authority`), классы мёржим через `cn`.
5. Экспортировать из барреля слоя (`components/ui/index.ts`, по алфавиту).
